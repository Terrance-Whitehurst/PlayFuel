## OQ-API-1 — Match Model Divergence

### Decision

**Hybrid (a)+(c).** Add 3 nullable text columns to `public.matches`; derive time-string fields at the iOS Repository/view layer.

### Verified Source State

- **DB (`0002_tables.sql`):** `public.matches` has `scheduled_start: timestamptz`, `estimated_duration_minutes: int`, `actual_end_at: timestamptz`, `surface: text`, `format: text`, `age_bracket: text`, `display_order: int`. **No** `round_label`, `opponent_label`, `court_label`.
- **iOS (`Match.swift`):** Has `scheduledTime: String`, `estimatedNextMatchTime: String?`, `round: String?`, `opponent: String?`, `court: String?`. No raw timestamp fields.
- **Gap confirmed:** 5 iOS fields have no DB column source.

### Rationale

- Parents at real tournaments record round/opponent/court — these are real-world inputs. Removing them is a product cost that isn't justified by implementation savings.
- Adding 3 nullable TEXT columns to the DB is a small migration with zero RLS changes needed (user ownership already enforced at the tournament level via cascade).
- `scheduledTime` and `estimatedNextMatchTime` are formatting concerns, not data concerns. `scheduled_start: timestamptz` is the authoritative source; timezone-aware string formatting belongs at the view/repository layer.
- `estimatedNextMatchTime` is derived from the *next* match in the tournament (sorted by `display_order` or `scheduled_start`), not stored separately.

### Downstream Actions

**Engineering (Backend / DB):**
- New migration `db/supabase/migrations/0005_match_labels.sql`:
  ```sql
  ALTER TABLE public.matches
    ADD COLUMN round_label    TEXT NULL,   -- e.g. "Round of 16", "Quarterfinal", freeform
    ADD COLUMN opponent_label TEXT NULL,   -- e.g. "Smith", "TBD"
    ADD COLUMN court_label    TEXT NULL;   -- e.g. "Court 7", "Stadium"
  ```
- Update `apps/api/src/playfuel_api/models/db.py` `MatchRow` Pydantic model: add `round_label: Optional[str] = None`, `opponent_label: Optional[str] = None`, `court_label: Optional[str] = None`
- Update `apps/api/src/playfuel_api/routes/matches.py` create/update request payloads to accept the 3 fields (all optional)
- Update `apps/api/README.md` endpoint inventory if request/response examples change

**Engineering (iOS):**
- Update `Networking/DTOs.swift` `MatchDTO` to decode `round_label`, `opponent_label`, `court_label` (snake_case → camelCase already handled by `snakeDecoder` with `.convertFromSnakeCase`)
- Update `MatchDTO.toModel()` to:
  - Format `scheduled_start` (timestamptz ISO 8601 string from API) → `scheduledTime: String` using a user-locale time formatter (e.g., `DateFormatter` outputting "9:00 AM")
  - Map `round_label` → `round`, `opponent_label` → `opponent`, `court_label` → `court`
  - Compute `estimatedNextMatchTime` at Repository layer (requires fetching all matches for the tournament, sorted by `display_order`, then taking `index + 1`'s `scheduled_start`); may be `nil` if no subsequent match exists
- Add a `DateFormatter+Match.swift` helper (or extension on `DateFormatter`) to encapsulate the time string formatting, keeping it testable and reusable

**Specs:**
- Update `.pi/multi-team/expertise/USER_STORIES.md` US-04 to note that `round`/`opponent`/`court` are user-entered annotations — no auto-population from any external provider in MVP
- No `RULES_CONSTANTS_V1.md` change needed — these fields are not consumed by the rules engine

**Effort estimate:** 1 backend session (migration + Pydantic + route update); piggyback iOS DTO changes onto Task #7 or a follow-up match-detail pass.

## New OQs Surfaced

### OQ-TRIAGE-1 — Timeline Block: iOS Model Shape vs Lead Spec (High — blocks Task #7 timeline implementation)

**Surfaced by:** PM source-verification pass before scribing.

**Finding:** The Lead's proposed `timeline` block JSON used field names and category values that do NOT match `apps/ios/PlayFuel/Sources/PlayFuel/Models/TimelineEvent.swift`:

| Aspect | Lead's Proposed Spec | iOS `TimelineEvent.swift` Actual |
|---|---|---|
| Title field | `label` | `title: String` |
| Category field | `category` | `kind: TimelineEventKind` (enum) |
| Match reference | `matchId: UUID?` | Not present |
| Detail field | Not specified | `detail: String` (required) |
| Event identity | Not specified | `id: UUID` (required, Identifiable) |

**iOS `TimelineEventKind` raw values (camelCase):** `wakeUp`, `meal`, `arrive`, `warmUp`, `match`, `recovery`, `hydration`

**Lead's proposed categories (snake_case):** `wake_up`, `breakfast`, `warmup`, `match_start`, `gap`, `food_window`, `pickup`, `match_end`, `recovery`

**Gap:** `gap`, `food_window`, `pickup`, `match_end` have no iOS `TimelineEventKind` equivalent. The mapping is non-trivial.

**Resolution path (decision needed before Task #7 implements timeline):**
- **(a)** API emits iOS-native `kind` values (`wakeUp`, `meal`, etc.) — no new iOS enum cases, but API must use camelCase enum values directly. Missing cases (`gap`, `food_window`, `pickup`) must map to an existing value or be dropped.
- **(b)** Extend iOS `TimelineEventKind` to add cases for `gap`, `food_window`, `pickup`, `match_end` — more semantic granularity, small iOS change.
- **(c)** `TimelineEventDTO` applies a mapping function: API snake_case → iOS camelCase, with a `.meal` fallback for unknown categories.

**Recommendation:** Option (b) — extend `TimelineEventKind` with the missing cases. These are semantically distinct events (gap window, food pickup window) that iOS views may want to style differently from generic `meal` events. Low-cost iOS change; no DB or API contract change.

**Owner:** Planning Lead (scope decision) + Engineering (iOS `TimelineEvent.swift` + API `timeline` spec alignment). Resolve before Task #7 scoping.
